# 数据清洗智能契约实施计划

## 目标

建立一套稳定的二期数据清洗契约，让报告生成链路从“按章节脚本临时识别字段”升级为“按契约取数、清洗、校验、降级、生成 PDF”。

最终目标：

1. 接口只负责取数和保存 raw JSON。
2. 清洗层统一读取 raw JSON，并按契约抽取每章需要的字段。
3. 每章生成稳定的 `chapterN_final_data.json`，再交给 Markdown/HTML/PDF 渲染。
4. 缺字段、口径不确定、单位异常时不编造，统一记录 warning 和降级策略。
5. 以后新增字段或调整口径，优先改契约表，不优先改生成器逻辑。

## 总体方案

采用“统一基础清洗 + 契约驱动取数 + 分章节业务清洗”的结构。

数据流：

```text
接口 getEmployeeIndexAiCcq
-> raw JSON 落盘
-> 通用基础清洗
-> 加载章节清洗契约
-> 按契约从 raw 指标行抽取字段
-> 分章节补充业务计算、排序、省略判断
-> chapterN_final_data.json
-> chapterN_final_report.md/html/pdf
```

核心原则：

1. 接口层不做业务清洗。
2. 通用清洗只处理结构、字段名、数字、单位、空值、路径索引。
3. 每章业务逻辑仍然独立，因为每章公式、排序、展示规则不同。
4. 契约表作为“字段需要什么、从哪里取、缺了怎么办”的唯一事实来源。

其中：

2. `chapter_cleaning_contract.md`：人读的字段总表，方便业务和开发校验。

4. `extractor.py`：按契约取数，输出字段值、命中来源和 warnings。
5. `validators.py`：统一校验必填字段、单位、数值范围、TOP 列表。

## 契约表设计

每个字段一条契约记录。建议 JSON 中每条记录包含：

```json
{
  "chapter_id": "chapter3",
  "chapter_title": "销量分析",
  "module": "3",
  "field_key": "sales_overview.month.actual",
  "field_name": "当月实际销量",
  "required_level": "required",
  "source_match": {
    "mode": "path_exact",
    "paths": ["三、销量分析-销量"],
    "metric_names": ["销量"],
    "date_type": "月"
  },
  "value_key": "实际值",
  "data_type": "number",
  "unit_policy": {
    "expected_units": ["万", "万元", "吨", ""],
    "target_unit": null,
    "convert": false
  },
  "missing_policy": "填 —，保留模板，不推断",
  "validation": {
    "allow_zero": true,
    "allow_negative": false
  },
  "usage": "3.1 销量概况表-实际行-当月列",
  "notes": "优先按 日期类型=月 命中；若接口日期类型为空，需要按同路径出现顺序兜底。"
}
```

字段解释：

| 字段 | 作用 |
| --- | --- |
| `chapter_id` | 对应章节，例如 `chapter2` |
| `module` | 来源接口模块 |
| `field_key` | 清洗后 JSON 的稳定字段路径 |
| `field_name` | 人看的字段中文名 |
| `required_level` | `required` / `optional` / `derived` / `degrade` |
| `source_match` | 如何从 raw 指标行命中字段 |
| `value_key` | 取 `指标数据` 中哪个字段，例如 `实际值`、`目标值`、`同期数` |
| `data_type` | `number` / `string` / `percent` / `amount` / `list` |
| `unit_policy` | 单位要求和换算规则 |
| `missing_policy` | 缺失时怎么处理 |
| `validation` | 值域和口径校验 |
| `usage` | 报告中使用位置 |
| `notes` | 口径备注 |

## 智能匹配规则

契约里的取数不要只靠 `指标名称`。同名指标很多，必须按优先级命中：

1. `指标路径` 精确匹配。
2. `指标路径` 前缀匹配。
3. `指标路径` 关键词组合匹配。
4. `指标名称 + 日期类型 + 单位` 辅助匹配。
5. 多条命中时按契约策略排序，无法唯一确定则记录 warning，不静默取第一条。

匹配结果必须保留来源：

```json
{
  "value": 123.4,
  "unit": "万元",
  "source": {
    "module": "2",
    "指标路径": "二、利润概况-营业收入（不含税）",
    "指标名称": "营业收入（不含税）",
    "value_key": "实际值"
  },
  "warnings": []
}
```

## 通用基础清洗

通用清洗输入：接口 raw response。

通用清洗输出：标准指标行列表。

标准指标行建议结构：

```json
{
  "module": "2",
  "month": "202606",
  "chapter_name": "二、利润概况",
  "指标名称": "营业收入（不含税）",
  "指标路径": "二、利润概况-营业收入（不含税）",
  "实际值": 6.826,
  "目标值": 0,
  "同期数": 0,
  "扣分值": -6.826,
  "达成率": 0,
  "权重分数": 0,
  "单位": "万元",
  "日期类型": "年",
  "raw": {}
}
```

通用清洗职责：

1. 递归清理字段名空格。
2. 校验外层 `code/message/data/章节数据`。
3. 展平 `指标数据` 到标准指标行。
4. 数字字符串转数值，保留原始 raw。
5. 统一处理百分号、逗号、空字符串。
6. 按 `module + 指标路径 + 指标名称 + 日期类型 + 单位` 建索引。
7. 输出 warnings，不中断可降级字段。

## 分章节业务清洗

### 第二章：利润概况

报告需要：

1. 营业收入。
2. 毛利率。
3. 毛利额。
4. 个人人工费用。
5. 基本薪酬 + 提成奖金 + 年终奖。
6. 差旅费。
7. 减值损失。
8. 财务费用。
9. 样板物料。
10. 其他各类费用。
11. 分摊前利润。
12. 当月、本季度累计、累计三个时间维度。

清洗动作：

1. 按指标路径和日期类型抽取利润瀑布表。
2. 统一金额单位，默认展示万元。
3. 计算费用合计、费用收入比。
4. 缺毛利率/毛利额时保留 `—`，不反推，除非契约明确允许公式。
5. 输出 `chapter2_final_data.json`，供 Markdown/PDF 使用。

### 第三章：销量分析

报告需要：

1. 销量概况表：目标、实际、达成率、达成差额、同比增长率、同比差额。
2. 时间维度：当月、本季度累计、累计。
3. 过程指标：招商生效客户、有效落地项目、打样项目数、产销客户数、产销项目数、客均销量、单项目销量。
4. 产品：销量金额、销售量吨数。
5. 客户：客户数、客均销量。
6. 项目：项目数、单项目销量。
7. 行业：行业销量、行业销量占比。

清洗动作：

1. 将同一指标按日期类型合并到同一字段组。
2. 目标值为 0 或缺失时，不计算达成率。
3. 同期数为 0 或缺失时，不计算同比。
4. 客户/项目模板目标 20/100 单独写入派生字段。
5. 产品、行业 TOP 排序优先用累计金额或占比。
6. 缺项目、行业、客户明细时保留模板位置和缺失说明。

### 第四章：毛利率与产品结构

报告需要：

1. 个人毛利率。
2. 产品收入占比。
3. 产品均价。
4. 价格方向。
5. 结构方向。
6. 收入占比 TOP3。
7. 均价上升/下降产品中收入占比 TOP3。

清洗动作：

1. 按产品名称合并 `%` 和 `元/KG` 两类记录。
2. `%` 写入收入占比。
3. `元/KG` 写入产品均价。
4. 方向字段优先使用接口方向；没有方向时暂按 `扣分值` 推断。
5. 同产品同单位多条记录时保留第一条并 warning。
6. 清洗结果保留每个产品的 raw_items，方便追口径。

### 第五章：应收分析

报告需要：

1. 个人应收款项总额。
2. 应收款项树。
3. 逾期金额前五客户。
4. 次月新增到期款前五客户。
5. 当年增加减值损失。
6. 减值损失影响金额 TOP5 客户。
7. 次月预计跳账龄 TOP5 客户。
8. 财务费用、利息支出、利息收入、资金占用费拆分。
9. 财务费用排名前三客户。

清洗动作：

1. 应收总额低于 10 万，整章省略。
2. 元转万元后再判断省略。
3. 总额缺失或单位不明时不省略，记录 warning。
4. 客户级列表按金额降序，最多取 TOP N。
5. 当前接口未上线完整客户级字段时，允许使用标准 fixture/适配层结构。

## 风险与处理

| 风险 | 处理 |
| --- | --- |
| 接口字段同名太多 | 必须使用完整 `指标路径`，不要只按 `指标名称` |
| 日期类型缺失 | 契约里写兜底策略，并记录 warning |
| 目标值或同期数为 0 | 不直接解释为 0%，除非契约确认该字段真实有效 |
| 单位混用 | 在契约中声明目标单位和换算规则 |
| 字段缺失 | 保留模板位置，输出 `—`，final data 记录 warning |
| 业务口径未确认 | 标记 `confirm`，不输出强结论 |
