# SimpleIntellReportV2 文件管理手册

本文档用于约束二期项目的代码与资料归档方式。后续开发、新对话接续或多人协作时，优先阅读并遵守本文件。

## 核心原则

1. 章节归章节，公共能力归公共层。
2. 不把某一章的生成逻辑、提示词组织、数据格式化代码放到其他章节文件中。
3. 接口请求、数据清洗、章节生成、报告保存、开发文档分别放在对应目录。
4. 只有被多个章节复用的能力才抽到公共模块；只服务单章的逻辑留在该章节文件内。
5. 临时数据、mock 数据、缺失数据说明必须可追踪，不混入正式生成逻辑。

## 推荐目录职责

```text
SimpleIntellReportV2/
├── Data/
│   ├── fetch_data.py              # 接口请求、批量拉取、API adapter
│   ├── check_data.py              # 数据基础检查
│   ├── contracts/                 # 各章节数据清洗契约
│   ├── normalize.py               # 通用数据清洗/标准化；需要时再新增
│   └── fixtures/                  # mock/sample 数据；需要时再新增
│
├── ReportGenerator/
│   ├── chapter1_generator.py      # 第一章：绩效得分与预警
│   ├── chapter2_generator.py      # 第二章：利润概况
│   ├── chapter3_generator.py      # 第三章：销量分析
│   ├── chapter4_generator.py      # 第四章：毛利率与产品结构
│   ├── chapter5_generator.py      # 第五章：行销行为
│   └── report_manager.py          # 报告目录、保存、调度等公共管理逻辑
│
├── guidelines/
│   ├── chapter1_guideline.md
│   ├── chapter2_guideline.md
│   └── chapter3_guideline.md
│
├── tests/
│   ├── test_chapter1.py
│   ├── test_chapter2.py
│   └── fixtures/                  # 测试专用样例数据；需要时再新增
│
├── Reports/                       # 正式报告输出
├── test_output/                   # 测试与验证输出
├── tmp/                           # 临时中间文件
│
└── DevelopmentDocs/               # 开发文档、接口问题、阶段说明
```

## 各目录规则

### Data

- 只放数据获取、接口适配、数据标准化、数据校验等逻辑。
- 不放 LLM prompt、不放章节正文生成逻辑。
- API key 不应硬编码在业务函数中，优先通过参数或环境变量传入。
- 等接口数据未补齐时，可以新增 mock/fixture 数据，但必须放在 `Data/fixtures/` 或 `tests/fixtures/`，并在文件名中体现用途。

### ReportGenerator

- 每个章节一个独立 generator 文件，例如第二章只能放在 `chapter2_generator.py`。
- 第一章代码不得写入第二章文件，第二章代码也不得写入第一章文件。
- 章节内可以包含该章节专属的数据格式化函数、prompt 拼装、章节生成流程。
- 如果某个格式化函数只服务第二章，保留在 `chapter2_generator.py`；不要提前抽到公共模块。
- `report_manager.py` 只做报告目录创建、文件保存、章节调度等管理工作，不写具体章节分析逻辑。

### guidelines

- 每章的写作要求、分析口径、提示词素材单独建文件。
- 命名使用 `chapter{n}_guideline.md`。
- 不同章节不要共用一个大 guideline，除非已经确认是全局通用规则。

### tests

- 测试文件与章节对应，例如第二章测试放 `test_chapter2.py`。
- 测试样例数据优先放到 `tests/fixtures/`，避免在多个测试文件中复制大段数据。
- 如果临时内联样例数据，必须保持规模可读，且只服务当前测试。

### DevelopmentDocs

- 放接口文档沉淀、数据缺失清单、开发约定、阶段性分析报告。
- 不放可执行业务代码。
- 数据缺失、字段口径不确定、接口返回异常等问题，要优先记录到该目录下的文档中。

### Reports

- 只放正式报告、准生产报告或需要对外交付的报告产物。
- 每次接口生成的正式结果按章节、员工、月份命名，例如 `chapter4_api_86002542_202606/`。
- 报告目录内可以保留 `raw/`，用于追踪当次报告对应的接口原始数据。
- mock、realtest、规则验证、AI 行动项保留等过程版本，不放在 `Reports/` 根下。

### test_output

- 放测试运行产物、人工验证产物、渲染样例和过程版本报告。
- 报告类验证统一放到 `test_output/report_validation/`。
- 这类文件可以用于复盘和对比，但不作为正式交付结果。

### tmp

- 放临时拆解、PDF 渲染图片、docx 模板解析文本等中间文件。
- 临时文件不应被业务代码长期依赖。
- 如果某个临时结果需要长期复用，应整理到 `DevelopmentDocs/`、`Data/fixtures/` 或 `test_output/`。

## 章节开发流程

1. 确认接口模块与章节映射。
2. 在 `Data/` 中完成接口拉取与基础校验。
3. 在对应的 `ReportGenerator/chapter{n}_generator.py` 中实现章节格式化和生成。
4. 在 `guidelines/chapter{n}_guideline.md` 中维护该章写作规则。
5. 在 `tests/test_chapter{n}.py` 中添加对应测试。
6. 如发现接口字段缺失或口径不清，记录到 `DevelopmentDocs/`。

## 禁止事项

- 禁止把第二章生成逻辑放到 `chapter1_generator.py`。
- 禁止把第一章、第三章等其他章节逻辑塞进 `chapter2_generator.py`。
- 禁止把接口请求代码散落到各章节 generator 中。
- 禁止把临时调试输出、一次性脚本、接口样例直接混入正式生成代码。
- 禁止把 mock、realtest、规则验证报告长期放在 `Reports/` 根目录。
- 禁止在不确认复用价值时提前创建过度抽象的公共模块。

## 新对话接续提示

如果重新开启新对话，请先搜索并阅读：

- `DevelopmentDocs/文件管理手册.md`
- `DevelopmentDocs/2023.md`
- `DevelopmentDocs/缺失数据清单.docx`
- 当前正在开发章节对应的 `ReportGenerator/chapter{n}_generator.py`
- 当前正在开发章节对应的 `guidelines/chapter{n}_guideline.md`
- 当前正在开发章节对应的 `tests/test_chapter{n}.py`

开发时以本手册作为文件归属判断依据；如果本手册没有覆盖，再结合现有代码结构就近放置，并补充本文档。
