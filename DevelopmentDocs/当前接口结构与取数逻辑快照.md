# 当前接口结构与取数逻辑快照

快照日期：2026-06-29  
用途：作为后续接口变更核对入口。详细字段级映射仍以下方“详细字段表”中的各章文档和当前代码为准，本文件记录当前接口外壳、公共字段、取数入口和各章清洗逻辑摘要。

## 1. 接口入口

### 1.1 章节指标接口

- 默认地址：`https://apidev.skshu.com/test/skshu-bi-api/biapitoxt/getEmployeeIndexAiCcq`
- 环境变量覆盖：`SKSHU_EMPLOYEE_INDEX_CCQ_API_URL`
- API key 来源：入参 `api_key` 优先，其次 `SKSHU_BI_API_KEY`
- 请求方式：`POST`
- API key 传递方式：URL query，`?apikey=<key>`
- 请求头：`Content-Type: application/json; charset=utf-8`
- 请求体：

```json
{
  "ZEMPLOYEE": "{区域经理工号}",
  "CALMONTH": "{接口取数月，YYYYMM}",
  "MOUDLE": "{模块号字符串，1-8}"
}
```

注意：字段名当前就是 `MOUDLE`，不是 `MODULE`。取数月 `CALMONTH` 与正文展示月份 `report_period` 分离；默认 `report_period = CALMONTH` 的上一个月，例如 `CALMONTH=202606` 默认展示 `202605`，正文显示为 `2026年1-5月`。

### 1.2 人员名单接口

- 默认地址：`https://apidev.skshu.com/test/skshu-bi-api/biapitoxt/getAiEmployeeOrgGcq`
- 环境变量覆盖：`SKSHU_EMPLOYEE_ORG_GCQ_API_URL`
- API key 来源同章节指标接口
- 请求方式：`POST`
- 请求体：`{}`

人员名单用于批量选择人员、补充报告元数据，并按工号去重统计第一章排名分母。

## 3. 人员名单字段映射

人员名单接口成功响应包含 `code/message/data/timestamp/executeTime`，其中 `data[]` 行字段当前为：

| 接口字段 | 内部字段 | 用途 |
| --- | --- | --- |
| `CALMONTH` | `calmonth` | 过滤当前月份人员 |
| `ZEMPLOYEE` | `job_id` | 区域经理工号 |
| `ZEMP_CD` | `sale_name` | 区域经理姓名 |
| `ZGWMC` 或 `岗位名称` | `sale_class` | 岗位名称，当前接口样本不一定提供 |
| `ZORGWB3` | `business_department` | 事业部 |
| `ZORGWB4` | `region` | 区域 |
| `ZORGWB5` | `province` | 省区 |
| `ZORGWB6` | `city_operation_department` | 城市经营部 |

`add_ranking_population_totals` 会按 `job_id` 去重后统计：

- `province_ranking_total`：同 `province` 人数
- `business_ranking_total`：同 `business_department` 人数

全量报告生成时，这两个分母会注入第一章排名记录，用于 `排名/总人数` 展示。

## 4. 接口门禁

公共校验函数：`Data/check_data.py::check_chapter_response`

当前规则：

- 响应必须是非空 dict。
- 若含 `error`，直接视为请求失败。
- `code` 允许 `None`、`1`、`"1"`；其他值失败。
- `data` 必须是非空 dict。
- `data.章节数据` 必须是非空 list。
- 每条 `章节数据` 必须是非空 dict，且至少有 `指标名称` 或 `指标路径`。
- `指标数据` 必须是非空 dict。
- 可按调用方要求校验章节名称关键字和 `指标数据` 必需字段。

单章脚本中，第二、三章通常要求 `指标数据.实际值` 和 `指标数据.日期类型`；第四章要求 `指标数据.实际值` 和 `指标数据.单位`。全量严格入口先拉 1-7 章，再按 `CHAPTER_KEYWORDS` 做章节名称门禁；`run_full_report_strict.py` 里第二、四章空数组有特定兜底判断，批量入口当前不放宽空数组。

## 5. 全量报告取数流程

核心入口：`ReportGenerator/full_report_generator.py::FullReportGenerator`

当前主链路：

1. 根据单人 `person_config`、`calmonth` 和 `report_period` 创建报告目录。
2. 并发请求 `SOURCE_CHAPTERS = range(1, 8)`，也就是 1-7 章；默认单人章节并发为 6。
3. 保存原始响应：
   - `raw/all_raw_chapters.json`
   - `raw/module_1.json` 到 `raw/module_7.json`
4. 逐章生成基础章节：
   - 每章先注入人员元数据。
   - 第一章额外注入人员名单统计出的排名分母。
   - 第四章会读取第二章年口径毛利率作为上下文。
   - 第六章会接收 `context_raw_data`，用于费用行动指南与其他章节联动。
   - 输出 `cleaned/chapterN_cleaned.json` 和 `markdown/chapterN.md`。
5. 第八章不在全量主链路中请求接口，而是调用 `build_chapter8_source(cleaned_by_chapter=1..7)` 派生：
   - `cleaned/chapter8_source.json`
   - `cleaned/chapter8_derived.json`
   - `markdown/chapter8.md`
6. 可选 AI 文案链路会一次生成并回填第 3、5、7、8 章文案；第 4 章当前仍是规则模板。结构化数字仍来自规则清洗结果。
7. 合并 1-8 章 Markdown，执行报告月份审计，输出完整 Markdown、HTML、PDF 和 `diagnostics.json`。

## 6. 各章取数逻辑摘要

### 6.1 第一章：绩效得分与预警，`MOUDLE=1`

当前代码：`ReportGenerator/chapter1_generator.py`

- 主要字段由 `CHAPTER1_FIELD_MAP` 定义。
- 多数字段按 `指标名称` 匹配，必要时叠加 `指标数据.日期类型`。
- 排名字段使用 `省区排名` 和 `部门排名`；`部门排名` 当前映射为事业部内排名。
- 绩效、销量、利润、季度奖金、年终利润等字段直接取 `指标数据` 中的 `实际值/目标值/达成率/扣分值/单位`。
- 季度奖金包含计算项，例如预计奖金、潜在逾期合计、同比下降阈值；来源和公式写入 `field_sources`。
- 未达百项目按 `指标路径` 包含“未达百绩效项目”筛选，排除 `月平均得分/全年总扣分` 等不可稳定识别项；能识别项目名时再展示。
- 清洗输出主键：`metadata/performance_score/sales/profit/quarter_bonus/year_end_profit/underperforming_items/field_sources/warnings`。

详细字段见：`DevelopmentDocs/第一章接口字段映射表.md`

### 6.2 第二章：利润概况，`MOUDLE=2`

当前代码：`ReportGenerator/chapter2_generator.py`

- 严格按 `指标名称 + 指标路径 + 指标数据.日期类型` 唯一匹配。
- 11 个利润科目，每个科目固定取月、季、年三个口径，共 33 个数据位。
- 取值字段固定为 `指标数据.实际值`，单位取 `指标数据.单位`。
- 不使用数组顺序、数值大小、相似名称或单位猜测。
- 除毛利率外不做单位换算；毛利率按客户确认口径执行 `实际值 * 100` 后展示为 `%`。
- 清洗输出主键：`metadata/cells/warnings`。

详细字段见：`DevelopmentDocs/第二章接口字段映射表.md`

### 6.3 第三章：销量分析，`MOUDLE=3`

当前代码：`ReportGenerator/chapter3_generator.py`

- 标准化为 `MetricRecord`，保留 `指标名称/指标路径/日期类型/实际值/目标值/同期数/扣分值/达成率/单位/客户编码/客户名称`。
- 主表和过程指标按 `指标名称 + 指标路径 + 日期类型` 精确命中。
- 目标、实际、扣分值、达成率尽量直接取接口字段。
- 打样项目数、产品/客户/项目/行业的增长率或差额按 `实际值` 与 `同期数` 计算。
- 产品、行业等 Top 逻辑只在明确有产品/行业名称的记录上做筛选排序。
- 清洗输出主键：`rows/stats`。

详细字段见：`DevelopmentDocs/第三章接口字段映射表.md`

### 6.4 第四章：毛利率与产品结构，`MOUDLE=4`

当前代码：`ReportGenerator/chapter4_generator.py`

- 只依赖完整 `指标路径`、`指标名称` 和 `指标数据.单位`。
- 当前主要证据类别：
  - `四、毛利率与产品结构-各产品的均价差异-<产品名>`
  - `四、毛利率与产品结构-各产品收入占比差异-<产品名>`
- 同名同路径不同值视为冲突，不按数组顺序或数值大小猜测。
- 全量报告中会从第二章提取年口径毛利率作为上下文传入。
- 清洗输出主键：`metadata-like fields/metric_evidence/analysis/calculations/conflicts/missing_fields/warnings`。

详细字段见：`DevelopmentDocs/第四章接口字段映射表.md`

### 6.5 第五章：应收分析，`MOUDLE=5`

当前全量代码：`ReportGenerator/chapter5_generator.py`  
当前单章严格脚本：`ReportGenerator/chapter5_strict.py`

全量报告使用 `chapter5_generator`：

- 支持三类输入：完整接口响应、接口 `data` 对象、内部结构化 mock。
- 若存在结构化字段，优先使用 `receivable_tree/overdue_top_customers/next_month_due_top_customers/impairment_summary/financial_expense` 等内部契约字段。
- 若只有 `章节数据[]`，则按指标路径清洗为应收树、次月到期、减值损失、账龄跳转、财务费用等结构。
- 应收总额用于省略判断：换算为万元后，低于 10 万元可省略第五章。
- TOP 客户类只有在客户名称、客户编码或可识别客户字段存在时才可稳定展示；否则保留待补充或 warnings。
- 清洗输出主键：`metadata/receivable_tree/next_month_due_total/overdue_top_customers/next_month_due_top_customers/impairment_summary/impairment_top_customers/aging_jump_top_customers/financial_expense/financial_expense_top_customers/raw_items/warnings`。

单章严格脚本使用 `chapter5_strict`：

- 按 `指标名称 + 指标路径` 精确匹配 `CHAPTER5_FIELD_MAP`。
- 财务费用相关字段按 Word 模板需要执行 `万元 * 10000 = 元`。
- 重复冲突时报告显示“待补充”，并在核对清单中保留候选证据。

详细字段见：`DevelopmentDocs/第五章接口字段映射表.md`

### 6.7 第七章：行销行为，`MOUDLE=7`

当前代码：`ReportGenerator/chapter7_generator.py`

- 支持 `章节数据[]` 和旧内部契约两种输入。
- 从指标行清洗时，当前不是完全严格路径匹配，而是按候选指标名、包含关键词、排除关键词和单位综合查找。
- 主字段：
  - `total_visit.actual/target/achievement_rate/deduction_score`
  - `project_visit.actual/target/achievement_rate/deduction_score`
  - `time_allocation.project_ratio/customer_ratio`
- 总拜访与项目拜访通常取单位 `次` 的实际/目标；达成率取单位 `%` 的达成率记录，或从同一记录 `指标数据.达成率` 兜底。
- 时间分配优先找项目/客户占比；缺失时会从“时间分配”且单位 `%` 的记录中兜底。
- 清洗输出主键：`metadata/total_visit/project_visit/time_allocation/warnings`。

契约文件见：`Data/contracts/chapter7_cleaning_contract.json`

### 6.8 第八章：总结

当前全量代码：`ReportGenerator/chapter8_source_builder.py` 和 `ReportGenerator/chapter8_generator.py`

- 全量报告当前不请求 `MOUDLE=8`。

- 单章 `run_chapter8_api.py` 仍支持请求 `MOUDLE=8` 或 fixture，但这不是全量报告主链路。

## 10. 后续接口变更核对清单

接口变更后优先核对：

1. 章节接口是否仍使用 `POST + ?apikey= + MOUDLE`。
